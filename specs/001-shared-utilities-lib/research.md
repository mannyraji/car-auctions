## R-01: npm Workspaces Monorepo Layout

**Decision**: npm built-in workspaces (`"workspaces": ["packages/*"]`) in root `package.json`. No Lerna, Turborepo, or pnpm.  
**Rationale**: CI expects plain `npm ci` + `npm run --workspaces --if-present <script>`. Adding a build orchestrator would require CI changes and is out of scope. npm workspaces is sufficient for a single shared library consumed by 8 sibling packages.  
**Alternatives considered**: Turborepo (adds caching but requires `turbo.json` and CI plugin), pnpm workspaces (uses `workspace:*` protocol instead of `*`, incompatible with npm CLI).  
**Implementation notes**:
- Root `package.json`: `"private": true`, `"workspaces": ["packages/*"]`. No build/lint/test scripts at root level — CI uses `--workspaces --if-present` flag to delegate to each package.
- Workspace packages reference each other via `"@car-auctions/shared": "*"` in dependencies.
- `npm ci` installs all workspace packages and hoists shared devDependencies to root `node_modules`.

## R-02: TypeScript Config — `module: Node16` vs Alternatives

**Decision**: `module: Node16` / `moduleResolution: Node16` in `tsconfig.base.json`, inheriting to all packages.  
**Rationale**: Constitution Pillar V Rule 4 explicitly mandates `module: Node16`. This is the correct choice for Node.js 20 ESM without a bundler.  
**Constraint identified**: With `module: Node16` and `"type": "module"` in package.json, all relative imports in TypeScript source MUST include `.js` extensions (e.g., `import { x } from './errors.js'`). TypeScript resolves `.ts` files at compile time but requires the `.js` extension in source to satisfy the Node16 module resolver at runtime.  
**Alternatives considered**: `moduleResolution: Bundler` (avoids `.js` extension requirement but requires a bundler — not present in CI), `module: CommonJS` (rejected per constitution).

## R-03: `@modelcontextprotocol/sdk` Transport Support

**Decision**: Use `StdioServerTransport` and `SSEServerTransport` from the SDK (v1.29.0). Implement a custom `WebSocketServerTransport` class using the `ws` package (v8.x).  
**Rationale**: The MCP SDK v1.29.0 natively provides stdio and SSE. WebSocket is NOT bundled. The spec (FR-011) and the gateway-mcp's real-time bid streaming explicitly require WebSocket. A custom `Transport` interface implementation wrapping `ws` is ~80 LOC and has no library coupling.  
**API surface** (`@modelcontextprotocol/sdk` v1.29.0):
```typescript
import { McpServer } from '@modelcontextprotocol/sdk/server/mcp.js';
import { StdioServerTransport } from '@modelcontextprotocol/sdk/server/stdio.js';
import { SSEServerTransport } from '@modelcontextprotocol/sdk/server/sse.js';
// WebSocket: custom implementation using ws
```
**Transport selection** (via `TRANSPORT` env var or `McpServerOptions.transport`):
```
stdio      → StdioServerTransport
sse        → SSEServerTransport (requires express, PORT env var)
websocket  → custom WebSocketServerTransport (requires ws, WS_PORT env var)
```
**SSE notes**: Requires an Express HTTP server. The `createMcpServer` helper creates the express app internally when `transport=sse`.

## R-04: VIN Decoder — NHTSA vPIC API

**Decision**: Use the NHTSA vPIC API (`https://vpic.nhtsa.dot.gov/api/vehicles/DecodeVin/{vin}?format=json`). Free, no authentication.  
**Rationale**: Spec assumption confirms "The NHTSA vPIC API remains free and publicly accessible without authentication." The API returns up to 140+ variable fields; only a subset is extracted (year, make, model, trim, engine type, body class, drive type, fuel type, transmission).  
**Caching strategy**: `better-sqlite3` ^12.8.0, WAL mode, table `vin_cache(vin TEXT PRIMARY KEY, result TEXT, cached_at INTEGER)`. TTL = 90 days (7_776_000 seconds). Prepared statements only.  
**Validation**: `validateVin(vin)` — `vin.length === 17 && /^[A-HJ-NPR-Z0-9]{17}$/i.test(vin)` (excludes I, O, Q via character class gap).  
**Error handling**: If NHTSA is unreachable, return `ToolResponse` with `success: false`, `error.code: 'SCRAPER_ERROR'`, `error.retryable: true`. No throw.

## R-05: Token Bucket Rate Limiter

**Decision**: Pure TypeScript token bucket, no npm dependencies. Continuous-refill model.  
**Rationale**: Zero external dependencies required (NFR-002). ~60 LOC implementation. Continuous refill avoids the thundering-herd problem of discrete interval timers.  
**Algorithm**:
- State: `tokens: number`, `lastRefillTime: number`
- `refill()`: `tokens = min(capacity, tokens + (elapsed / interval) * tokensPerInterval)`
- `consume(n)`: async, queues callers when `tokens < n`, wakes them after computed delay
- Default: capacity=1, interval=3000ms (1 token per 3 seconds)
**Note**: The `TokenBucket` class is internal to `priority-queue.ts` — NOT exported from the public API.

## R-06: Priority Queue — Starvation Prevention Algorithm

**Decision**: Age-based effective-priority promotion with a periodic check interval (500ms).  
**Rationale**: Each task carries a `createdAt` timestamp. A background timer boosts `effectivePriority` when `waitTime > maxWait[priority]`. Queue is re-sorted after each promotion cycle. FIFO within the same effective priority level.  
**Priority levels and guarantees**:

| Level      | Base value | Max wait  | Spec guarantee                    |
|------------|-----------|-----------|-----------------------------------|
| critical   | 100       | 0 ms      | Immediate (bypasses ordering)     |
| high       | 75        | 2 000 ms  | Dequeued within 2s under light load |
| normal     | 50        | 5 000 ms  | Dequeued within 5s under light load |
| low        | 25        | 10 000 ms | Dequeued within 10s under light load |
| background | 0         | 30 000 ms | ≥ 1 slot per 60s under heavy load |

**Critical bypass**: Critical tasks are moved to the front of the pending list and bypass sorting. They still call `rateLimiter.consume()` before execution (cannot exceed 1 req/3s).  
**`npm p-queue` evaluated and rejected**: `p-queue` v8 supports concurrency and intervals but not multi-level priorities with starvation prevention. Rolling own is ~200 LOC and zero dependency.

## R-07: OpenTelemetry Bootstrap Strategy

**Decision**: Use `@opentelemetry/sdk-node` ^0.214.0 + `@opentelemetry/exporter-trace-otlp-http` ^0.214.0. No-op when `OTEL_EXPORTER_OTLP_ENDPOINT` is unset.  
**Rationale**: Spec FR-017. The SDK auto-instruments Node.js internals by default; this feature uses manual instrumentation only (no auto-instrumentation) to avoid unexpected overhead.  
**Bootstrap pattern**:
```typescript
// called once at process start
export function initTracing(serviceName: string): void {
  if (!process.env.OTEL_EXPORTER_OTLP_ENDPOINT) return; // no-op path
  const exporter = new OTLPTraceExporter();
  const sdk = new NodeSDK({ traceExporter: exporter, serviceName });
  sdk.start();
}
```
**Span wrapper**:
```typescript
export async function withSpan<T>(
  name: string,
  attrs: SpanAttributes,
  fn: () => Promise<T>
): Promise<T> {
  const tracer = trace.getTracer('car-auctions');
  return tracer.startActiveSpan(name, async (span) => {
    const start = Date.now();
    try {
      const result = await fn();
      span.setAttributes({ ...attrs, 'tool.status': 'ok', 'tool.duration_ms': Date.now() - start });
      return result;
    } catch (err) {
      span.setStatus({ code: SpanStatusCode.ERROR, message: (err as Error).message });
      span.setAttributes({ 'tool.status': 'error', 'tool.duration_ms': Date.now() - start });
      throw err;
    } finally {
      span.end();
    }
  });
}
```
**No-op guarantee**: When `OTEL_EXPORTER_OTLP_ENDPOINT` is unset, `initTracing` returns immediately. `withSpan` still calls `trace.getActiveSpan()` / `tracer.startActiveSpan` but with the default no-op tracer provider — zero allocation overhead.  
**Error safety**: OTLP export failures are swallowed by the SDK batch exporter; they do not propagate to application code (spec edge case: "OTEL endpoint unreachable must fail silently").

## R-08: `playwright-extra` + Stealth Plugin Compatibility

**Decision**: `playwright` ^1.59.1 + `playwright-extra` ^4.3.6 + `puppeteer-extra-plugin-stealth` ^2.11.2.  
**Rationale**: `playwright-extra` wraps the standard Playwright launcher and adds plugin support. The stealth plugin works with both `puppeteer-extra` and `playwright-extra` via a shared plugin interface. Versions confirmed compatible as of 2026-04.  
**Integration pattern**:
```typescript
import { chromium } from 'playwright-extra';
import StealthPlugin from 'puppeteer-extra-plugin-stealth';
chromium.use(StealthPlugin());
const browser = await chromium.launch({ proxy: { server: process.env.PROXY_URL ?? '' } });
```
**Browser pool max concurrency**: Default 3 contexts (NFR-004). Configurable via `BrowserPoolOptions.maxContexts`.

## R-10: Vitest Config for Node16 ESM

**Decision**: Per-package `vitest.config.ts`. `environment: 'node'`. Import `@car-auctions/shared` source directly via path alias during tests (no pre-build required).  
**Rationale**: Vitest handles TypeScript natively via `esbuild` transform — no `ts-node` needed. The `resolve.alias` in `vitest.config.ts` maps `@car-auctions/shared` to `./src` so that intra-package tests and future cross-package tests do not require a prior `npm run build`.  
**Coverage**: Vitest v8 provider, threshold 80% branches on `src/**` excluding `src/index.ts`.
